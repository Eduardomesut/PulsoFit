# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

PULSO (PulsoFit) — a Spanish-language single-page web app that generates a personalized weekly diet plan from a short questionnaire (goal, sex, age, weight, height, diet type, disliked foods, allergies, meals per day). Every meal comes from a structured recipe catalog with a photo, ingredients with quantities, and step-by-step preparation instructions. All UI text, variable names, and comments in the codebase are in Spanish; keep new code consistent with that convention.

## Commands

```bash
npm install        # install dependencies
npm run dev         # dev server at http://localhost:5173
npm run build       # tsc -b && vite build (type-checks, then builds to dist/)
npm test            # vitest run — pure logic + recipe-catalog invariant tests
npm run test:watch  # vitest in watch mode
npm run preview     # preview the production build
```

Tests (Vitest) live next to the code in `src/*.test.ts` and cover only the pure logic in `src/logica.ts` — no component/DOM tests yet. The deploy workflow runs `npm test` before building, so a failing test (including a catalog-invariant violation) blocks deployment. There is no linter configured; `npm run build` (which runs `tsc -b` first) is the way to check for TypeScript errors.

## Architecture

- **`src/main.tsx`** — entry point; wraps `<App>` in `AuthProvider`.
- **`src/logica.ts`** — data catalogs and all pure (React-free) logic, imported by `PulsoFit.tsx` and by the tests:
  - Image/text helpers: `U` (Unsplash URL builder), `youtubeUrl`, `normalizar` (lowercase + strip accents). `FOODIMG` is the image bank (ingredient-type key → Unsplash id); each recipe references a `FOODIMG` key via its `img` field.
  - `TIPOS_DIETA` / `ALERGENOS` / `ALIMENTOS` — option catalogs for the preference questions (diet type, allergies, disliked foods). `OBJETIVOS` — the four goals.
  - `RECETAS` — ~50 structured recipes (`id`, `nombre`, `categoria`, `img`, `kcalAprox`, `ingredientes` with quantities, `pasos` preparation steps, `dietas`, `alergenos`, `contiene`, `objetivos`). Authoring rule: per category keep ≥5 vegan, ≥5 gluten-free, ≥5 lactose-free and ≥4 "safe" recipes (vegan + zero allergens) so filtering never empties a category — enforced by `src/catalogo.test.ts`.
  - `RECETAS_CINE` — iconic dishes from series/movies; never enter the weekly plan or the PDF. Optional `fotoEscena` (real still from the show, hotlinked from Fandom wikis / Wikimedia) and `fotoPlato` (real dish photo from Wikimedia Commons); both fall back to `FOODIMG` on load error via the `conRespaldo` handler in `FichaReceta`.
  - `REPARTO` — meal slots + calorie split per meals-per-day (3/4/5); `filtrarRecetas` — filters `RECETAS` by allergies + diet type (strict) and disliked foods (relaxed if a category drops below 4 recipes); `buildDiet` — deterministic 7-day plan from the filtered pools.
  - `calcularMetricas` — Mifflin-St Jeor calorie/macro calculation per objetivo (`perder` / `equilibrio` / `ganar` / `musculo`).
  - `migrarDatos` — migrates saved plans from the old fitness-era format (maps old objetivo ids, drops `nivel`/`dias`, fills in new preference fields).
  - Community-recipes logic: `ADMIN_EMAIL` (the only account allowed to delete other users' recipes — must match the email hardcoded in the `recetas_comunidad` RLS delete policy), `puedeBorrarReceta` (author-or-admin, only for rows with `user_id`; decides button visibility, RLS enforces), `INGREDIENTES_WEB` (the "despensa": unique ingredient names derived from `RECETAS`, the only ingredients allowed in community recipes), `CATEGORIAS_RECETA`, `validarRecetaComunidad` (Spanish error list for a draft; empty = publishable).
  - `resumenCatalogo` / `resumenUsuario` — compact plain-text context (recipe list without ingredients/steps; user preferences) sent by the Chef IA chat to keep token cost low.
- **`src/logica.test.ts` / `src/catalogo.test.ts` / `src/comunidad.test.ts`** — Vitest suites: pure functions, recipe-catalog invariants (unique ids, valid vocabularies, authoring rule, no diet+allergy combination leaves a category under 4 recipes), and community-recipes logic (despensa, draft validation, delete permissions).
- **`src/PulsoFit.tsx`** — styling constants and every screen component. There is no router; `App` (in this file) drives a simple state machine via `fase` (`"hero" | "form" | "scan" | "plan" | "recetario" | "cine" | "actualidad" | "restaurantes" | "favoritos" | "crear" | "chef"`) and renders one screen at a time. Key pieces inside this file, top to bottom:
  - Style constants and image fallback helpers (`FALLBACK_IMG`, `onImgError` — inline SVG fallback when a photo fails to load). The visual language is retro food-truck (inspired by pizza-amici.nl): `C` palette (cream `bg`, navy `line`/`text` used for ink AND borders, mustard `hot2` for actions, tomato red `hot1` accents, light-blue `panel2`), `DF` display font (Anton, always uppercase), `MONO` body font (Azeret Mono), `SERIF` (Instrument Serif italic) for handwritten-style captions, hard offset shadows (`4px 4px 0 navy`) instead of blurs, sticker-style buttons that lift on hover and squash on press (`.btn-cta`), slightly rotated hero badge, and a looping `Marquesina` strip. `grad`/`gradText` are kept as legacy names but now resolve to flat mustard / red. Fonts load from Google Fonts in `index.html`.
  - Animations use **GSAP** (`gsap` + `ScrollTrigger`, registered once at module load) and **Lenis** (momentum smooth scroll), both free deps. `useScrollSuave` (called once in `App`) boots Lenis and wires it to GSAP: `lenis.on("scroll", ScrollTrigger.update)`, feeds `lenis.raf` from `gsap.ticker`, and `lagSmoothing(0)`. Lenis's base CSS is imported in `main.tsx` (`lenis/dist/lenis.css`). The module-level `REDUCE` flag (from `prefers-reduced-motion`) short-circuits Lenis and every GSAP effect so reduced-motion users get instant scroll and no animation.
  - Animation helpers: `Aparece` (wraps content, reveals it with a GSAP `back.out` pop + rotation via `ScrollTrigger` when it scrolls into view; used for the long recipe lists in `Recetario`/`Cine` — pass `delay`/`rot` for staggering) and `FotosFlotantes` (hero-only floating polaroids from `POLAROIDS`; the outer `.polaroid` is driven by `gsap.quickTo` for damped cursor parallax + proximity repulsion off a `mousemove` listener, while the inner `.polaroid-marco` levitates on the CSS `flotar` loop — two different elements so the transforms don't clash; hidden below 1080px). Both use `gsap.context(...).revert()` for cleanup. Remaining pure-CSS keyframes/classes in the global `<style>` in `App`: `flotar`, `.polaroid`/`.polaroid-marco`, `popIn`, `fadeUp`, `kenburns`, `scanline`, `marquesina`.
  - `App` — root component and state machine.
  - `AuthModal` — login/signup modal (uses `useAuth` from `src/auth.tsx`).
  - `Hero` → `Formulario` (5-step questionnaire: objetivo, about you, diet type, preferences, meals/day) → `Scan` (loading transition) → `Plan` (final results screen with expandable recipe cards showing ingredients + preparation, PDF export, "save plan" for logged-in users).
  - `BotonCartel` — placard/sign-style menu link that bounces with GSAP: hover gives an `elastic.out` pop (scale + small random rotation + growing hard shadow), press squashes then re-bounces. `variante` `"chip"` (compact, desktop header) or `"bloque"` (full-width, mobile menu); `activo` fills it mustard. Sets inline `transition: none` so GSAP owns its transform (avoids fighting the global `button` CSS transition). No-op under reduced motion. Used for the nav links in both `Cabecera` (desktop) and `MenuLateral` (mobile).
  - `Cabecera` — shared top bar: navy `Marquesina` ticker strip plus a white sticker-style bar (navy border + hard shadow) with logo left, centered `BotonCartel` nav links, contextual `acciones` right, and `MenuLateral` slide-over panel on narrow screens via the "Menú" button. `MenuLateral` is driven by a GSAP timeline (backdrop fade + panel slide-in + staggered `.menu-enlace` links); closing runs the timeline in reverse (via `tl.reverse()` + `onReverseComplete`) before unmounting, so every close goes through its `cerrar` handler, not `onCerrar` directly. Reduced-motion skips the animation both ways. Simple hover/press states (buttons, cards) stay as CSS transitions by design — GSAP is only for entrance/scroll/parallax choreography. When logged in and a plan is available (current `datos` or `planGuardado`), a "Mi rutina" link appears first (provided by App through `RutinaCtx`, highlighted when `actual="rutina"`) and jumps back to the `Plan` screen from anywhere. Catalog sections are reachable from it on every screen (App remembers the origin screen in `seccionDesde`): `Recetario` (all recipes incl. cine ones, searchable by name/ingredient — accent-insensitive via `normalizar` — with category filter chips), `Cine` (series/movie dishes), `Restaurantes` (city/area search that embeds a keyless Google Maps iframe — `output=embed`, no API key — with cuisine-type chips preselected from the user's `tipoDieta`) and, only when logged in, `Favoritos`. The desktop bar keeps only `Recetario` and `Restaurantes` as direct placards; `Cine`, `Actualidad` and `Favoritos` hang from the `MenuCategorias` dropdown ("Categorías" placard, opens on hover or click, GSAP timeline: sticker panel pops in and its placards drop staggered with an elastic bounce, reversed on close; the placard highlights when `actual` is one of its sections). The mobile `MenuLateral` uses the flat `enlacesNav(user)` list instead. All build on the shared `CabeceraSeccion` (banner header), `FichaReceta` (expandable card, scene banner when `fotoEscena` exists, optional `onBorrar` delete button) and `DetalleReceta` (ingredients + steps + YouTube link, also used by `Plan`).
  - Community recipes: `Recetario` fetches `recetas_comunidad` rows (newest first, "👥 Comunidad" filter chip, author shown in the card label) and shows a create CTA when auth is `enabled` (login prompt if logged out); `CrearReceta` (fase `"crear"`, reached only from Recetario) is the publish form — ingredients are picked from `INGREDIENTES_WEB` only, draft validated by `validarRecetaComunidad` before the insert. Delete buttons appear per `puedeBorrarReceta` and rows are removed optimistically after the Supabase delete succeeds; RLS is the real gate.
  - Favorites: `FavoritosCtx` (filled by `App` only when logged in with Supabase configured; `null` otherwise) exposes `{ ids, alternar }` — the user's favorite recipe ids (`favoritos` table, optimistic toggle with revert on error). When the context is non-null, `FichaReceta` shows a heart sticker over the thumbnail (a sibling of the header button — buttons can't nest), so hearts appear in Recetario/Cine/Actualidad/Favoritos automatically. `Favoritos` (fase `"favoritos"`, nav link only when logged in) lists the saved recipes by filtering `recetasUnificadas` (the shared community+catalog+cine+actualidad mapping, also used by `Recetario`; community rows come from the shared `useRecetasComunidad` hook); ids that no longer resolve (deleted community recipes) are silently ignored.
  - `descargarPDF` (inside `Plan`) — lazy-loads `jspdf` only when the user requests a PDF; includes the weekly plan plus a deduplicated "RECETARIO" section with full recipes. Uses a `san()` helper that strips glyphs jsPDF's standard fonts can't render (em-dash, ×, ·, etc.).
- **`src/auth.tsx`** — `AuthProvider`/`useAuth`: thin wrapper around Supabase auth (sign in/up/out, session state). Exposes `enabled` (whether Supabase is configured) so the rest of the app can degrade to guest mode.
- **`src/supabase.ts`** — creates the Supabase client from `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` env vars. If either is missing, `supabase` is `null` and `isSupabaseConfigured` is `false` — **the app must keep working in guest mode (no login) when Supabase isn't configured**; don't assume `supabase` is non-null anywhere without checking.
- **`supabase/schema.sql`** — full Postgres schema/RLS policies for the Supabase project; run manually in the Supabase SQL editor (not migrated automatically). Currently defines `perfiles` (one row per user, auto-created via trigger on signup), `planes` (one saved plan per user, keyed by `user_id`), `recetas_comunidad` (user-published recipes: public select, insert own rows only, delete own rows or any row if the JWT email equals the admin email — keep that literal in sync with `ADMIN_EMAIL` in `src/logica.ts`), `favoritos` (one row per user + recipe id; `receta_id` is text because it mixes catalog ids and community uuids; each user only sees/writes their own rows) and `chef_usos` + `consumir_uso_chef()` (atomic daily-quota counter for the Chef IA — the limit is enforced in the DB, not the client). Future phases (per the file's own comments and `SUPABASE_SETUP.md`) will add `registros` (progress tracking) and `amistades` (friends).
- **`supabase/functions/chef/index.ts`** — Deno Edge Function backing the `Chef` screen (fase `"chef"`, a logged-in-only cooking chat): validates the Supabase JWT, consumes the daily quota (10/day) via `consumir_uso_chef`, then proxies a streaming call to the Claude API (`claude-haiku-4-5`, capped `max_tokens`) with the cooking-only system prompt built server-side; the SSE body is passed through untouched and `x-usos-restantes` reports the remaining quota. The `ANTHROPIC_API_KEY` secret lives only here. Deploy steps in `SUPABASE_SETUP.md` §7. The front (`Chef` component) sends the recent message tail plus a compact context built from `resumenUsuario`/`resumenCatalogo` and today's plan, and degrades gracefully when Supabase or the function are missing.

## Data flow / persistence

- Guest mode: everything lives in React state (`datos` in `App`); nothing persists across reloads.
- `datos` shape: `{ objetivo, sexo, edad, peso, altura, tipoDieta, alergias[], noGusta[], comidasDia }`. The generated plan is never stored — it's rebuilt deterministically from `datos` via `buildDiet`/`calcularMetricas`.
- Logged-in: on login, the app fetches the user's saved plan (`planes` table, one row per `user_id`, upserted), runs it through `migrarDatos` (old fitness-era rows are converted; unrecognizable objetivos disable resume), and offers to resume it ("Continuar con mi plan guardado"). RLS policies restrict each user to their own `perfiles`/`planes` rows.

## Deployment

Static site deployed to GitHub Pages via `.github/workflows/deploy.yml` on push to `main` (or manual dispatch). `vite.config.ts` sets `base: "/PulsoFit/"` for production builds vs `"/"` in dev — keep this in sync if the repo/Pages path ever changes. Supabase credentials are injected at build time from GitHub Actions repository **variables** (not secrets, since the anon key is public by design) — `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`. If unset, the build still succeeds and the deployed app runs in guest mode.
